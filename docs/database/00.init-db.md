# Database

Raw SQL over [db0](https://db0.unjs.io) (Nitro's `useDatabase` connector layer), Postgres dialect. No ORM.

- **Local dev:** PGlite (Postgres in WASM) at `.data/pg` — no Docker, zero setup.
- **Production:** any Postgres via `DATABASE_URL` (db0's `postgresql`/node-postgres connector) — e.g. Neon's pooled connection string on Vercel.
- Driver picked by env: `DATABASE_URL` unset → PGlite; set → node-postgres.
- **Better Auth** bypasses db0: a `pg` Pool in production, a small Kysely dialect over the same PGlite instance in dev (`server/utils/pglite-dialect.ts`). Columns use Better Auth's camelCase defaults, so `@better-auth/cli generate` output works as-is.

> This is a **template**. Migrations in `server/database/migrations/` are demo-only — squash
> them into your own `0000` once you pick a real database; there's no data to preserve.

## Key files

| File                                      | Role                                                                     |
| ----------------------------------------- | ------------------------------------------------------------------------ |
| `server/database/migrations/*.sql`        | Schema source of truth: plain SQL, applied in filename order.            |
| `server/database/migrate.ts`              | Runner: `_migrations` tracking table, one transaction per file.          |
| `server/database/migrate-cli.ts`          | `pnpm db:migrate` entry; picks the connector from `DATABASE_URL`.        |
| `server/utils/db.ts`                      | `useDb()` (db0 connector per env) and `useAuthDatabase()` (Better Auth). |
| `server/utils/pglite-dialect.ts`          | Dev-only Kysely dialect over the shared PGlite instance.                 |
| `server/database/dev-migration-plugin.ts` | Dev-only auto-migrate, wired in `vite.config.ts` (`nitro({ plugins })`). |

## Workflow

No schema-diffing tool — migrations are written by hand:

```bash
# 1. add server/database/migrations/000N_short-name.sql with your DDL
# 2. dev server auto-applies it on save/restart (dev plugin)
pnpm db:migrate    # apply to DATABASE_URL (prod / CI deploy step); PGlite fallback locally
```

Querying uses db0's tagged template (values are bound as parameters):

```ts
const db = await useDb()
const { rows } = await db.sql<{ rows: Todo[] }>`
  SELECT * FROM "todos" WHERE "userId" = ${userId} ORDER BY "createdAt" DESC`
```

Adding a Better Auth plugin with its own tables? `npx @better-auth/cli generate` prints the
SQL for the missing tables — paste it into a new migration file.

## Deploy (Vercel)

1. Vercel → Marketplace → **Neon** integration → injects `DATABASE_URL` (use the pooled `-pooler` connection string).
2. Build command: `pnpm db:migrate && pnpm build`.
3. Same `postgres://` works on any Node host — swap providers by changing the URL.

## Switching database engine

- **Swap provider, same dialect** (Neon → Supabase → RDS → self-hosted Postgres): change `DATABASE_URL` only.
- **Swap engine/dialect** (Postgres ↔ SQLite/libSQL/MySQL): swap the db0 connector in `server/utils/db.ts`, port the `.sql` migrations, and give Better Auth the matching handle (its Kysely adapter covers postgres/mysql/sqlite/mssql).

## Picking a migration strategy

This template stays agnostic on purpose — plain SQL migrations run on any db0-supported
engine with no ORM lock-in. In practice, most real projects are better served by picking an
ORM with a proper migration toolchain. [Drizzle](https://orm.drizzle.team) is a solid default:
typed schema, `drizzle-kit generate`/`migrate`, and a [Better Auth adapter](https://www.better-auth.com/docs/adapters/drizzle).

To migrate this template to Drizzle, hand Claude this prompt:

```
Migrate this template's database layer to Drizzle ORM + drizzle-kit (Postgres dialect):
`pnpm remove db0 kysely && pnpm add -D drizzle-orm drizzle-kit`.
Recreate `server/database/schema.ts` from `server/database/migrations/*.sql` (https://orm.drizzle.team/docs/sql-schema-declaration, https://orm.drizzle.team/docs/column-types/pg) and add `drizzle.config.ts` (https://orm.drizzle.team/docs/drizzle-config-file) — `dialect: 'postgresql'`, driver `pglite` when `DATABASE_URL` is unset.
Replace `useDb()`/`useAuthDatabase()` in `server/utils/db.ts` with a single `useDrizzle()`: `drizzle-orm/node-postgres` in prod (https://orm.drizzle.team/docs/get-started/postgresql-new), `drizzle-orm/pglite` in dev (https://orm.drizzle.team/docs/get-started/pglite-new), keeping the `import.meta.env.DEV` gate that dead-code-eliminates PGlite from prod builds.
Wire Better Auth through its drizzle adapter (https://www.better-auth.com/docs/adapters/drizzle) and delete `server/utils/pglite-dialect.ts`.
Replace `server/database/migrate*.ts` with drizzle-kit generate/migrate (https://orm.drizzle.team/docs/kit-overview) and `drizzle-orm/pglite/migrator` in the dev plugin. The existing migrations are demo-only: regenerate them from scratch with `drizzle-kit generate` rather than porting.
Update `db:*` scripts, the tests in `server/`, and `docs/database/`.
```
