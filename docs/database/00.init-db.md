# Database

Raw SQL over [db0](https://db0.unjs.io) (the connector layer behind Nitro's `useDatabase`), **Postgres** dialect. No ORM.

- **Local dev:** PGlite (Postgres in WASM) at `.data/pg` — no Docker, zero setup.
- **Production:** any Postgres over `DATABASE_URL` via db0's `postgresql` connector (node-postgres) — Neon's pooled connection string on Vercel, or any other provider/host.
- Driver is chosen by env: `DATABASE_URL` unset → PGlite; set → node-postgres.
- **Better Auth** does not go through db0: it gets a `pg` Pool in production and a tiny Kysely dialect over the _same_ PGlite instance in dev (`server/utils/pglite-dialect.ts`), and runs its built-in Kysely adapter on top. Column names are Better Auth's camelCase defaults, so `@better-auth/cli generate` output works as-is.

> This is a **template**. The migrations in `server/database/migrations/` are demo-only
> (they just prove dev + prod work). When you pick your database, you can squash them into
> your own `0000` and start fresh — there is no real data to preserve.

## Touch-points (the files any DB swap edits)

| File                                      | Role                                                                                                                                            |
| ----------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| `server/database/migrations/*.sql`        | The schema's source of truth: plain SQL files, applied in filename order.                                                                       |
| `server/database/migrate.ts`              | Migration runner: `_migrations` tracking table, one transaction per file. Dialect-agnostic as long as the connector's instance can run scripts. |
| `server/database/migrate-cli.ts`          | `pnpm db:migrate` entry (runs TS directly with `node`). Picks the connector from `DATABASE_URL`.                                                |
| `server/utils/db.ts`                      | `useDb()` — picks the db0 connector per env — and `useAuthDatabase()` — the handle Better Auth runs on. Row types (`Todo`) live here too.       |
| `server/utils/pglite-dialect.ts`          | Dev-only Kysely dialect over the shared PGlite instance (what the `kysely-pglite` package does, without its CLI/codegen dependency tree).       |
| `server/database/dev-migration-plugin.ts` | Dev-only auto-migrate. Registered only in dev via `nitro({ plugins })` in `vite.config.ts`, so it never ships to prod.                          |
| `.env.example`                            | The connection env var(s).                                                                                                                      |
| `package.json`                            | DB driver deps (`db0`, `pg`, `@electric-sql/pglite`, `kysely`).                                                                                 |

## Workflow

There is no schema-diffing tool: write migrations by hand.

```bash
# 1. add server/database/migrations/000N_short-name.sql with your DDL
# 2. dev server auto-applies it on save/restart (dev plugin)
pnpm db:migrate    # apply to DATABASE_URL (prod / CI deploy step); PGlite fallback locally
```

Querying uses db0's tagged template (values are bound as parameters):

```ts
const db = await useDb()
const { rows } = await db.sql<{ rows: Todo[] }>`
  SELECT * FROM "todos" WHERE "userId" = ${userId} ORDER BY "createdAt" DESC`
```

Adding a Better Auth plugin with its own tables? `npx @better-auth/cli generate` prints the
SQL for the missing tables — paste it into a new migration file.

### Taking over an existing drizzle database

The runner baselines databases migrated by this template's previous drizzle-kit setup: if
`drizzle.__drizzle_migrations` exists and `_migrations` is empty, `0000_init.sql` (the
squashed drizzle history) is recorded as applied without running, and later migrations
apply normally. Fresh databases just run everything.

## Deploy (Vercel)

1. Vercel → Marketplace → **Neon** integration → it injects `DATABASE_URL` (use the pooled `-pooler` connection string).
2. Build command: `pnpm db:migrate && pnpm build`.
3. Same `postgres://` works on any Node host — swap providers by changing the URL.

## Switching database engine

- **Swap provider, same dialect** (Neon → Supabase → RDS → self-hosted Postgres): change `DATABASE_URL` only. No code change.
- **Swap engine/dialect** (Postgres ↔ SQLite/libSQL/MySQL): swap the db0 connector in `server/utils/db.ts` (db0 has `libsql`, `mysql2`, `sqlite`, … connectors), port the `.sql` migrations to the new dialect, and give Better Auth the matching handle (its built-in Kysely adapter supports postgres/mysql/sqlite/mssql). The runner in `migrate.ts` only assumes "instance can execute a SQL script".
